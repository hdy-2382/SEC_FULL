# 지표 핸드북 — 원리 · 산식 · 적용법

[PROCESS.md](PROCESS.md)가 "언제 무엇을 하나", [CRITERIA.md](CRITERIA.md)가 "어떤 잣대로 판정하나",
[RECORD_SCHEMA.md](RECORD_SCHEMA.md)가 "무엇을 기록하나"라면, 이 문서는 **화면의 각 지표가
어디서 온 개념이고, 왜 그 단계에서 쓰며, 어떻게 읽고, 어떻게 오용되는지**를 정리한다.
원형(업계 표준)과 우리 응용(변형·간이화)을 구분해 적는다 — 변형을 표준인 척하지 않기 위해.

## 1. 뼈대 프레임 2개

### FRACAS — 고장 보고·분석·시정 체계
- **원형**: Failure Reporting, Analysis & Corrective Action System. 미 국방 신뢰성 관리
  (MIL-HDBK-2155 계열)에서 정립. 핵심 원리: **고장은 사건이 아니라 데이터**이며,
  보고→분석→시정→**검증**의 루프가 닫혀야 신뢰성이 성장한다.
- **우리 적용**: 공통 레코드 스토어(`records[]`)가 FRACAS 대장이다. 단계별로 강도만 다르다 —
  POC=FRACAS-lite(필수 5필드), Pilot=full(버전·무발생검증 필수), 운영=필드 FRACAS(알람 승격 기준).
  화면의 폐루프 패널(신규→조치중→검증중→종결)이 이 루프의 상태 기계.
- **오용 경계**: "조치 완료"로 끝나면 FRACAS가 아니다 — **무발생 검증 통과까지가 한 사이클**.
  재발(§5) 추적이 없는 FRACAS는 조치 일지에 불과하다.

### Stage-Gate — 단계·게이트 프로세스
- **원형**: 신제품 개발 관리 표준(R.G. Cooper). 기간(단계)에는 데이터를 쌓고,
  게이트에서 **사전 확정된 EXIT CRITERIA**로 통과/보류를 판정한다.
- **우리 적용**: POC→Pilot→양산 시범→[인증 심의]→확산→운영의 사다리. 잣대 사전 확정·사후
  변경 금지(3원칙 ③)가 게이트의 신뢰를 만든다. 심의(이관·투자)는 기간이 아니라 이벤트 —
  새 데이터를 만들지 않고 기존 증거를 심사한다(체크리스트).

## 2. 시간·사이클 지표 (신뢰성 정량)

| 지표 | 산식 | 원리 | 우리 적용 | 주의 |
|---|---|---|---|---|
| **MTBF / MCBF** | 누적 가동(h/Cy) ÷ 고장 수 | 지수분포 가정의 평균 고장 간격 | Pilot부터 (성장곡선), 양산평가·운영 상시 | **POC 금지** — 표본 소수+설계 유동이면 평균은 소음. "통계 금지"의 근거 |
| **MTTR** | Σ복구시간 ÷ 건수 | 정비성(Maintainability) | 첫 측정은 POC 비정상 평가(의도 주입), 운영 SLA(≤10분 등) | 복구 정의(자동/수동) 사전 합의 |
| **가동률(A)** | 가동시간÷계획시간 ≈ MTBF/(MTBF+MTTR) | Availability | 운영 월간 RAM 헤드라인 (목표 98% 등) | 계획 정지 제외 규칙 명문화(CRITERIA §1) |
| **RAM** | 위 셋의 묶음 | Reliability·Availability·Maintainability | 운영/관제 월간 보고 프레임 | — |

### MCBF 성장곡선 (Pilot)
- **원형**: 신뢰성 성장 시험(TAAF: Test, Analyze And Fix)과 Duane/Crow-AMSAA 성장 모델.
- **우리 적용**: 간이형 — 주차 누적 MCBF(누적 Cy ÷ 누적 고장)를 꺾은선으로. 보는 것은 절대값이
  아니라 **기울기**(수렴하는가)와 목표선 도달 여부.
- **주의**: 수정 배포(버전) 없이 오르는 곡선은 우연이다 — 버전 이력과 함께 읽는다.

## 3. 무고장 런과 신뢰수준 입증

- **연속 무고장 런**: 실증 사다리(사외 72h 무에러 → 사내 300h 무정지 → 1개월 → 호기별 SAT 축약 런).
  **원형**: success-run(무고장 시험) — 무고장 n Cy가 입증하는 신뢰수준은
  `C = 1 − e^(−n/MTBF목표)` (CRITERIA §6). 예: 목표 MTBF 100Cy를 80% 신뢰수준으로 입증하려면 161Cy.
- **리셋 규칙**: 위반 시 0부터 재시작, 부분 인정 없음. **잣대의 설득력은 리셋에서 나온다** —
  그래서 화면은 리셋(↺)을 숨기지 않고 표기한다.
- **Error Budget**: 계약 에러 한도(예: 완주 중 3회). 용어는 SRE에서 차용했지만 실체는
  **계약 파라미터 동결**(양산평가 M1) — 사후 협상 금지가 본질.

## 4. 우선순위 도구

- **Pareto**: 80/20 법칙 — 소수 고장모드가 다수 건을 만든다. 개발 단계는 **건수순**(수정개발
  우선순위), 운영 단계는 **다운타임(손실 시간)순** — 건수 Pareto와 순서가 다르면
  "드물지만 오래 세우는 고장"이 먼저다(비용 관점).
- **S×O 위험 매트릭스**: FMEA의 RPN(S×O×D)에서 검출도(D)를 뺀 간이형. 심각도(Critical/Major/Minor,
  결과 기준) × 발생도 밴드(드묾<3 ≤보통<6 ≤빈발) → High/Medium/Low 조치 우선순위.
  전 템플릿 공통(공통 레코드 기반).
- **FMEA**: POC P2에서 초판 — 이때 만든 **고장모드 어휘(코드마스터)가 전 단계 기록의 언어**가 된다.
  운영의 필드 고장모드는 차기 과제 FMEA로 환류(경험 자산화).

## 5. 분류 축과 재발 — 단계별 질문의 구현

| 단계 | 축 | 질문 |
|---|---|---|
| POC | 4분류: 컨셉 리스크/설계/구현(SW)/시험환경 | 컨셉의 병인가, 고칠 수 있는 병인가 |
| Pilot·양산 | 근본원인 6분류: 설계/부품/제작·조립/SW/시험환경·자재/운영·조작 | 어디를 고쳐야 하나 |
| 확산·운영 | 원인계층: 설계/제작·조립/설치·시공/운영·환경 | 전 함대의 병인가, 이 호기의 병인가 |

- 축은 **교체가 아니라 세분화**(상위호환) — POC '설계'가 Pilot에서 설계/부품/제작·조립으로 쪼개진다.
  그래야 단계를 넘는 통계가 이어진다.
- **재발**: 동일 고장모드(표준분류)의 재출현 — **종결 여부 무관**(CRITERIA §5). 재발 = 근본원인
  미해결 신호 → 재분석 의무. 재발률은 "땜질 vs 시스템"을 가르는 단 하나의 숫자.
- **무발생 검증**: 조치 후 동일 모드 무발생 N Cy(기본 `verifyCycle`) 경과 시에만 종결.
  §3 success-run의 축소 적용 — "고쳤다"의 통계적 최소 근거.
- POC의 **전수 4분류 보드**는 지표가 아니라 **결론 장치**다: 컨셉 리스크 0건 입증이 POC의 성적표.

### 5.5 고장모드 어휘의 운영 — 업체와 만들고, 단계와 함께 키운다

| 시점 | 업체 | PM | 합동 · 산출물 |
|---|---|---|---|
| POC 착수 (P2, 미팅 1회) | 개발 경험 기반 모드 후보 제안 | 유사 과제 어휘 지참(FMEA 환류분) | **어휘 v1 (10~20개)** — 명칭·정의·4분류 기준 합의 → 코드마스터 초안 |
| POC 운영 (데일리→주간) | 이슈로그 5필드 — **모드는 어휘에서 선택**, 새 유형은 '후보' 표기 | 4분류 승인 · 컨셉 리스크 판단 | 주간 어휘 리뷰 — **추가는 버전 이벤트**(v1.1), 소급 재분류 없음 |
| Pilot 이관 | — | 중복 통합 정리안 | **정식 코드 부여 + 등급 확정 + 4→6분류 매핑표** — 기존 레코드는 매핑으로 일괄 세분화(재분류 아님) |
| Pilot 운영 | 코드 선택 + SW/HW 버전 필수 | 조치검증 · 무발생 종결 판정 | 재발 시 재분석 리뷰 |
| 양산 시범 | 기록 + 증거 첨부 | 판정대장 | 관련/비관련 합동판정 — 어휘 사실상 동결 |
| 확산·운영 | (관제) 알람 승격 건만 코드 부여 | 계층 리포트 · CIP | 신규 필드 모드 → **차기 과제 FMEA 환류** |

축 세분화 매핑(사전 확정): 컨셉 리스크→(POC에서 0 입증 후 소멸) / 설계→설계·부품·제작·조립 /
구현(SW)→SW→(계층에선 설계로 — 전 함대 리스크) / 시험환경→시험환경·자재→운영·환경 /
운영·조작(Pilot 신설)→운영·환경·설치·시공(확산 신설).

**3대 규칙**: ① 자유 텍스트 금지(어휘에서 선택 — 재발 추적의 전제) ② 어휘 추가는 버전 이벤트
③ 세분화는 허용·임의 재분류는 금지(매핑표로만 — 통계 신뢰 보호).

**분류 판단 순서** (위에서부터, 처음 YES에서 멈춤):
- POC 4분류: Q1 이 아키텍처로는 해결 불가? → 컨셉 리스크(게이트 보류) / Q2 장비 밖 요인? → 시험환경
  / Q3 코드·파라미터만으로 해결? → 구현(SW) / 나머지 → 설계 개선. 컨셉 판단만은 PM+설계 리더.
- 확산 원인계층: Q1 다른 호기에서도 재현? (층화 분포가 단서) → 설계(전 함대 에스컬레이션)
  / Q2 설치·시공 상태 기인? → 설치·시공 / Q3 개체 제작 편차? → 제작·조립 / 나머지 → 운영·환경.

## 6. 판정·심의 도구

- **관련/비관련 합동판정 (판정대장)**: 원형은 신뢰성 시험의 relevant/non-relevant failure 분류
  (MIL-STD-781 계열). 공정 연결 후엔 외생 원인이 섞이므로 ①기준 사전 서면 합의 ②증거 패키지
  ③합동 리뷰 ④사후 재분류 금지 ⑤비관련도 시정 오너 지정(CRITERIA §4).
- **Known Issues Register (처분대장)**: 운영승인심의(ORR: Operational Readiness Review) 관행 —
  이관 시 오픈 건을 전건 처분(종결예정/carry-over 조건부 이관/waiver 위험수용)하고 기한·오너를
  서명한다. 우리 구현: 별도 문서가 아니라 **공통 레코드의 오픈 건 필터 + 처분대장 조인**.
- **CIP**: 지속 개선(Continuous Improvement). 운영 단계에서 다운타임 Pareto 상위 모드를
  개선과제로 닫고 효과를 검증한다.
- **8D**: 자동차 업계 표준 문제해결 절차 — 운영 만성(재발) 고장의 심층 분석에 적용.

### TECOP — 원형과 우리 응용 (자주 묻는 것)
- **원형**: 대형 프로젝트·에너지 업계(Shell 계열)의 **리스크 분류 프레임** —
  Technical / Economic / Commercial / Organisational / **Political**. 본래 용도는 리스크
  레지스터에서 리스크를 식별·태깅하는 축이지, 상태 신호등이 아니다.
- **우리 응용(변형)**: ① 게이트 리뷰 **고정 안건 체크**로 사용 — 기술 지표만으로 게이트가
  통과되는 것을 막는 장치(P1 "비기술 리스크 조기 차단"의 상시화). ② P를 Political →
  **안전·인허가(Permit/Safety)**로 재해석 — 사내 과제에서 실질 리스크는 정치가 아니라
  안전인증·법정 인허가 일정이므로. ③ 값은 자동 계산이 아니라 **PM의 판단**(양호/주의+비고).
- **오용 경계**: TECOP은 지표가 아니라 **안건**이다 — 점수화·합산하지 않는다. 각 렌즈의
  '주의'는 비고(무엇이 왜)와 함께여야 의미가 있다.

## 7. 한 장 요약 — 단계 × 지표 매핑

| 단계 | 핵심 질문 | 주 지표/장치 | 금지·주의 |
|---|---|---|---|
| 개발(제작) | 계획대로 만들어지나 | 마일스톤 진척 vs 일정 경과, 지연 건수 | 평가 지표 없음이 정상 |
| POC | 컨셉의 병인가 | 전수 4분류 보드, 72h 무에러 런, 비정상 평가(MTTR 첫 측정), 수렴 추이 | **통계(MTBF) 금지** |
| Pilot | 수렴하는가 | MCBF 성장곡선, Pareto, 재발 0, 시정조치 검증마감, 버전 기록 | 설계 동결 전 런 무효 |
| 양산 시범 | 계약 기준 이내인가 | 연속 Cy 완주, Error Budget, 관련/비관련 판정대장, 신뢰수준 입증 | 사후 재분류 금지 |
| 인증(심의) | 어떤 조건으로 안고 가나 | Known Issues Register(처분: 종결예정/carry-over/waiver) | 새 데이터 생성 금지 |
| 확산 | 전 함대 병인가 | 호기별 퀄(SAT+축약 런), 원인계층, 호기 층화, 설계성 에스컬레이션 | 기준 구성 동결 |
| 운영 | 어떤 고장부터가 경제적인가 | 월간 RAM, 알람→FRACAS 승격률, 다운타임 Pareto, CIP, 8D | 알람 전수≠FRACAS |
| 전 단계 공통 | — | 공통 레코드, 폐루프 상태, 재발 링크, S×O 매트릭스, TECOP | 합산 KPI 오독 주의 |
