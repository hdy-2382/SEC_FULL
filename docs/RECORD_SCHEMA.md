# 공통 고장 레코드 명세

**원칙: 고장/이슈 레코드의 스키마는 전 과제·전 단계 공통.** 단계가 오르면 필수 필드가 늘어날 뿐, 형식은 같다.
그래야 "Pilot의 정지가 POC 때부터 있던 에러의 재발"임을 추적할 수 있고, 과제 간 Pareto 비교가 가능하다.
POC에서 필수를 5개로 제한하는 이유: 초기 단계에 보고 부담을 지우면 기록 자체가 사라진다.

## 2. 심각도 정의

| 등급 | 기준 |
|---|---|
| **Critical** | 안전 관련, 또는 라인/장비 정지가 필요하고 즉시 복구 불가 |
| **Major** | 기능 저하·수동 개입 필요, 생산에 영향 |
| **Minor** | 자동 복구 또는 경미 개입, 생산 영향 미미 |

## 3. 원인분류 (단계별 축)

- **POC — 4분류** (이 단계 보고의 핵심): ① **컨셉 리스크** (이 아키텍처로 해결 불가 — 0건 입증이 POC의 결론) ② 설계 개선 ③ 구현(SW) 버그 ④ 시험환경 요인
- **Pilot·양산 시범 — 근본원인 분류**: 설계 / 부품 / 제작·조립 / SW / 시험환경·자재 / 운영·조작
- **확산·운영 — 원인 계층**: 설계 (→ 전 함대 리스크, 즉시 에스컬레이션) / 제작·조립 / 설치·시공 / 운영·환경

## 4. 엑셀 시트 매핑 (업체 보고 → 웹)

| 런기록 | POC | 일자·런시간(h)·에러수·비고 | 업체 |

### 4-1. 런 기록·일일평가 관리 규칙 (데이터 관리 항목)

런 게이지·주차 추이·MCBF가 전부 이 기록에서 파생되므로 **원장(元帳)으로 관리**한다.

- 필수 필드: **일자 · 런시간(h)(또는 일일평가 횟수) · 에러수** — 비고는 리셋·개선 반영 등 사건이 있는 날 기재
- 미가동일은 행을 생략한다 (0h 행 금지 — 가동 시간 통계 왜곡 방지)
- **기록 공백 금지**: 평가 기간 중 마지막 기록이 오늘 대비 **7일 이상** 뒤처지면 빌드가 경고한다
- 일자 누락·중복·역순 금지 — 빌드가 검증 (`_validate_runlog`)
- 에러수>0인 날 = 리셋 (docs/CRITERIA.md §3) — 사후 수정 금지

- 시트명·헤더는 변경 금지 (빌드가 별칭 사전으로 흡수하지만, 표준 준수가 원칙)
- 사진: 엑셀에 붙여넣지 않는다. `사진(파일명)` 칸에 파일명만 쉼표 구분으로 적고, 실물은 zip으로 전달 → `errors/`에 동일 파일명 배치
